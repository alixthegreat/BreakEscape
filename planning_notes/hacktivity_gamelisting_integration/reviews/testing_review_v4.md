### TEST-v4-1: Phase 1.7 Does Not Resolve TEST-v3-11 — Engine Dummy App Still Lacks VmSet/Vm Models

**Severity:** High
**Phase:** 1.7

**Finding:**
TEST-v3-11 explicitly recommended that Phase 1.6/1.7 either (a) test `vm_panel` and `vm_set_panel` only in the Hacktivity suite (where `VmSet` and `Vm` exist), or (b) add minimal stub models to the engine's `test/dummy/app/models/`. The Revision 5 Phase 1.7 spec adds a detailed set of test cases to `test/controllers/break_escape/games_controller_test.rb` — an **engine** test file — without addressing this constraint at all.

The engine's `test/dummy/` application loads only the engine's own models. `VmSet` and `Vm` are Hacktivity models. In the engine test environment:

- `defined?(::VmSet)` returns `nil`, so `vm_panel` and `vm_set_panel` return `head :not_found` immediately via the `hacktivity_mode?` guard — every happy-path test case in Phase 1.7 will receive a 404 regardless of fixture state, making the test vacuously pass the wrong assertion.
- The Phase 1.7 fixtures reference `@game.vm_set_id` — but in the dummy app there is no `VmSet` record to assign. Any test that sets `game.vm_set_id = some_fixture_id` persists a foreign-key value that points to a non-existent table.
- The authorisation cases ("Player who owns the game → proceeds normally") require the action to reach the `authorize` call. It never will under these conditions.

Revision 5 added a detailed test spec list but left the foundational environment problem unsolved. A developer implementing Phase 1.7 who runs the engine's test suite will see all 1.7 test cases passing green (they all receive 404, which the authentication/authorisation negative cases expect) while the positive-path logic is entirely uncovered.

**Recommendation:**
Resolve TEST-v3-11 explicitly as part of Revision 5. Choose one of the two options and commit to it in the plan text:

**Option A (recommended for speed):** State that all `vm_panel` and `vm_set_panel` tests live exclusively in `test/controllers/hacktivity/game_slots_controller_test.rb` (or `games_controller_test.rb` if these actions are in the engine's `GameSlotsController`). Remove Phase 1.7's reference to `test/controllers/break_escape/games_controller_test.rb` and move the full test list to Phase 4.6.

**Option B:** Add to `test/dummy/app/models/vm_set.rb` and `test/dummy/app/models/vm.rb` the minimal stub interface needed by these two actions:
```ruby
# test/dummy/app/models/vm_set.rb
class VmSet < ApplicationRecord
  has_many :vms
  belongs_to :sec_gen_batch, optional: true
end

# test/dummy/app/models/vm.rb
class Vm < ApplicationRecord
  belongs_to :vm_set
end
```
The dummy app then also needs a migration for these tables, a fixture file, and `hacktivity_mode?` stubbed to `true` in the test setup. The plan must specify all of this.

### TEST-v4-2: Phase 1.7 Redirect URL Assertions Cannot Verify Hacktivity Nested Routes in the Engine Test Environment

**Severity:** High
**Phase:** 1.7

**Finding:**
Phase 1.7 includes the test case:

> "Redirect URL contains correct vm_set and vm ids (not another player's)"

and:

> "Redirect URL contains `embedded=1`"

Both require the engine's test to assert on the value of a redirect URL that is generated by calling `Rails.application.routes.url_helpers.event_sec_gen_batch_vm_set_vm_path(...)` (per ARCH-v4-1, a hard-coded Hacktivity route). The engine's dummy app does not mount Hacktivity's routes. `event_sec_gen_batch_vm_set_vm_path` does not exist in the dummy app's `url_helpers`. Calling it raises `NoMethodError`, not a routing error — meaning the test will raise an exception during setup, not during the assertion, making the failure message opaque.

Even if the path helper were somehow available, the engine's tests cannot route back to Hacktivity URL structure without either (a) mounting Hacktivity as an engine inside the dummy app (heavy, fragile) or (b) stubbing `Rails.application.routes.url_helpers` (which is valid but must be specified in the plan).

The architecture review (ARCH-v4-1) recommends replacing the hard-coded route calls with configurable callbacks (`config.vm_panel_url_for`). If that recommendation is adopted, the engine's test can stub `BreakEscape.config.vm_panel_url_for` to return a predictable string and assert on that — completely decoupling the engine's test from Hacktivity's route structure. This is the correct resolution.

**Recommendation:**
Phase 1.7 must specify the stub strategy for URL generation. If ARCH-v4-1 is implemented (configurable callback), add to Phase 1.7 setup:

```ruby
setup do
  BreakEscape.configure do |config|
    config.vm_panel_url_for = ->(_game, vm_title) {
      "https://hacktivity.test/vms/#{vm_title}?embedded=1"
    }
  end
end
```

Then assert `assert_redirected_to "https://hacktivity.test/vms/kali?embedded=1"`. This verifies the redirect target without any dependency on Hacktivity's route table.

If ARCH-v4-1 is deferred, Phase 1.7 must instead specify that the redirect URL assertion is performed in the **Hacktivity** test suite only, and the engine's test only asserts that a redirect occurred (e.g. `assert_response :redirect`), not the URL content.

### TEST-v4-3: Phase 1.7 Missing Test for `Pundit::NotDefinedError` — Policy Methods Are Not Yet Specified

**Severity:** High
**Phase:** 1.7

**Finding:**
Phase 1.7 includes:

> "Player who does NOT own the game (other user's game id) → denied (Pundit 403 or redirect to root)"
> "Admin → allowed"

These assume `BreakEscape::GamePolicy#vm_panel?` and `BreakEscape::GamePolicy#vm_set_panel?` exist. As identified in ARCH-v4-5, these policy methods are not specified anywhere in the Revision 5 plan. Pundit's `authorize` call resolves to the policy class by convention and raises `Pundit::NotDefinedError` if the method is absent — this is not a 403 or a redirect, it is an unhandled exception producing a 500.

The Phase 1.7 authorisation tests will therefore:
- Either pass vacuously if the engine test environment never reaches `authorize` (because `vm_panel` returns 404 at the `hacktivity_mode?` guard — see TEST-v4-1).
- Or raise `Pundit::NotDefinedError` in the Hacktivity suite where `hacktivity_mode?` returns true, making the test error (not fail) with a misleading message.

There is no test case in Phase 1.7 that explicitly verifies the policy method exists and returns the correct value. The test list includes the **effect** ("denied") but not the **cause** (the policy method), making a missing method invisible until production.

**Recommendation:**
Add a policy unit test to Phase 1.7, placed in `test/policies/break_escape/game_policy_test.rb`:

- `vm_panel?` — owner with active game → `true`
- `vm_panel?` — non-owner → `false`
- `vm_panel?` — admin → `true` (via the `admin?` helper)
- `vm_panel?` — owner with completed game → `false` (per ARCH-v4-5 recommendation to guard on `record.status == 'active'`)
- Same four cases for `vm_set_panel?`

These are pure unit tests with no routing dependency and can live in the engine test suite without the VmSet stub problem.

Additionally, add a test case to Phase 1.7 controller tests:

- `vm_panel` called when `BreakEscape::GamePolicy#vm_panel?` would raise `NotDefinedError` (i.e. method absent) → verify the test setup includes the policy method, not that the action handles the exception.

### TEST-v4-4: `enable_console` Reset Fixture State Is Undefined — Tests Assert on Default That May Not Hold

**Severity:** High
**Phase:** 4.6

**Finding:**
Revision 5 adds to Phase 4.6:

> "After `start` (VM-backed): all VMs in assigned VmSet have `enable_console: false`"
> "After `restart`: all VMs reset to `enable_console = false`"

Both tests require a known starting state for `enable_console`. The plan does not specify:

1. **What `enable_console` is set to in the fixtures before the action fires.** If fixtures set `enable_console: false` by default, the `start` test proves nothing — the assertion will pass even if the code block that performs `update_all(enable_console: false)` is completely deleted. The test is only meaningful if the fixture VMs start with `enable_console: true` (simulating the SecGen-provisioned state where console access is open by default).

2. **The `restart` test has the same problem in the opposite direction.** The `restart` test is verifying that VMs are reset to `false` after a restart. If the test does not explicitly set `enable_console: true` on the fixtures before calling `restart`, the assertion that they are `false` afterwards is trivially true and does not verify the reset code path.

3. **The `extend_vms` second-call test** ("does NOT redispatch activate") is specified in the plan but the assertion mechanism is not: the plan says "`DispatchVmCtrlService` does NOT redispatch activate" without specifying a stub strategy. Without stubbing `DispatchVmCtrlService.ctrl_vm_async` (e.g. with `expects` from Mocha or a Rails mock), this assertion cannot be written. The plan should specify: "stub `DispatchVmCtrlService.ctrl_vm_async` with a call-count assertion; on the second `extend_vms` call, assert call count remains 0 for the `activate` command."

**Recommendation:**
Add fixture state annotations to Phase 4.6 for each `enable_console`-sensitive test:

```yaml
# test/fixtures/vms.yml
kali_vm:
  vm_set: player_one_vm_set
  title: kali
  enable_console: true   # <-- intentionally true to prove the reset path

web_server_vm:
  vm_set: player_one_vm_set
  title: server-web
  enable_console: true   # <-- same
```

And specify for the `start` test:

> "Setup: VMs in the assigned VmSet have `enable_console: true`. After `start`, reload all VMs and assert each `enable_console == false`."

For the `extend_vms` second-call test, specify the stub:

```ruby
DispatchVmCtrlService.expects(:ctrl_vm_async).never
# or with Minitest mocks:
mock_service = Minitest::Mock.new
# assert ctrl_vm_async is not called in the second extend_vms request
```

### TEST-v4-5: Phase 4.7 VNC Button Selector Is Unspecified — Tests Cannot Be Written

**Severity:** High
**Phase:** 4.7

**Finding:**
Phase 4.7 specifies:

> "`enable_console: true` → VNC/console controls rendered"
> "`enable_console: false` → VNC/console controls NOT rendered"
> "assert_no_match on VNC button selector"

The phrase "VNC button selector" is a placeholder. The plan does not specify:

- The actual HTML element (is it a `<button>`, `<a>`, `<form>`?).
- The CSS class, `id`, `data-*` attribute, or text content that identifies the VNC control.
- Whether `VmsController#show` renders an ERB partial (e.g. `_vnc_controls.html.erb`) that can be asserted on by partial name, or whether the element must be found by inspecting the rendered HTML.
- What `assert_no_match` is called on: `response.body`? A specific CSS selector via `assert_select`?

Without this specification, a developer writing the Phase 4.7 tests must reverse-engineer the existing `vms/show.html.erb` view to find the VNC element. This inverts the TDD intention: the test should be written against a known contract, not discovered from implementation. More practically, if the view renders the VNC button as `<%= link_to 'Open Console', vnc_url %>`, the test must know the link text or target; if it uses `data-action="open-vnc"`, the test uses `assert_select '[data-action="open-vnc"]'`.

**Recommendation:**
Phase 4.7 must specify the exact assertion pattern. Based on typical Hacktivity view conventions:

```ruby
# enable_console: true
assert_select 'a[href*="vnc"]', count: 1

# enable_console: false
assert_select 'a[href*="vnc"]', count: 0
# or: assert_no_match /vnc/, response.body
```

The plan should explicitly name the selector or link text being asserted, and note which partial or view file contains the VNC control. If the VNC control does not yet have a stable identifier, Phase 4.7 should specify adding `data-testid="vnc-button"` (or equivalent) to the view as a prerequisite.

### TEST-v4-6: Phase 4.7 Cross-Engine Tests Require `break_escape_games` Fixtures in Hacktivity — Not Specified

**Severity:** High
**Phase:** 4.7

**Finding:**
Phase 4.7 specifies these "BreakEscape cross-engine" test cases in `test/controllers/vms_controller_test.rb`:

> "BreakEscape cross-engine: Player's VM with `enable_console: false` → 200 but VNC button absent"

1. A `BreakEscape::Game` record in Hacktivity's test database that references the player and the `VmSet` being tested.

### TEST-v4-7: `vm_panel` Called on a Completed Game — No Test, Behaviour Undefined

**Severity:** Medium
**Phase:** 1.7 / 4.6

**Finding:**
The Phase 1.7 test spec covers authentication and authorisation but includes no lifecycle state cases. Specifically, there is no test for `vm_panel` called when `game.status == 'completed'`.

This is a plausible real-world scenario: a player completes the game, the `finish` action fires, the game is marked `completed` and `RelinquishVmSetJob` is enqueued. The player's browser tab still has the Phaser HUD open. If the player clicks a terminal button before the Phaser client detects the game-over state, `vm_panel` is called on a completed game.

The plan does not specify the expected response. Three plausible outcomes each have different implications:

1. **Policy guard (`BreakEscape::GamePolicy#vm_panel?` returns false for completed games):** Pundit raises `NotAuthorizedError` → redirect or 403. This is the ARCH-v4-5 recommended approach (`record.status == 'active'`). If implemented, a test should verify this.
2. **No guard:** `vm_panel` proceeds, calls `vm.update_column(:enable_console, true)` on a VmSet that may already be relinquished (deleted). If `VmSet.find_by` returns nil, the action returns 404. If the VmSet still exists (relinquishment is async), the action sets `enable_console: true` on a relinquished VmSet and redirects. Neither outcome is tested.
3. **`game.vm_set_id` is still set, VmSet is in the process of being relinquished:** The `VmSet.find_by` call returns the record, `enable_console: true` is set, and the player gets a live VNC URL to a VmSet in mid-relinquishment.

Without a test, all three outcomes are untested and the actual behaviour depends on the race between `vm_panel` and `RelinquishVmSetJob`.

**Recommendation:**
Add to Phase 1.7:

- `vm_panel` called when `game.status == 'completed'` → response is denied (403 / redirect), not 404. (This requires `GamePolicy#vm_panel?` to check `record.status == 'active'` per ARCH-v4-5.)
- `vm_panel` called when `game.status == 'in_progress'` but `VmSet.find_by` returns nil (VmSet already relinquished) → 404, no `enable_console` change.

### TEST-v4-8: `enable_console` Race Condition (ARCH-v4-4) Has No Test Coverage Specified

**Severity:** Medium
**Phase:** 1.7 / 4.6

**Finding:**
ARCH-v4-4 identified a race between `vm_panel`'s `vm.update_column(:enable_console, true)` and `restart`'s `vm_set.vms.update_all(enable_console: false)`. If both requests are processed concurrently, the single-row write from `vm_panel` may win after the bulk reset from `restart`, leaving a VM with `enable_console: true` in a restarted game.

The architecture review recommends guarding `vm_panel` with `vm.update_column(:enable_console, true) if @game.reload.status == 'active'`. If this mitigation is implemented (which it should be), there must be a test for it. No such test exists in Phase 1.7's test spec.

The plan currently lacks any test that exercises the interaction between `vm_panel` and `restart` — these actions are tested in separate test cases with no shared game state. A test that calls `vm_panel` after `restart` (in sequence, not truly concurrent) at minimum verifies that the policy/status guard works correctly. True concurrency testing is impractical in `ActionDispatch::IntegrationTest`, but the sequential post-restart `vm_panel` case is a meaningful regression guard.

**Recommendation:**
Add to Phase 1.7 (integration section):

- `vm_panel` called after `restart` has fired (game `status == 'active'` but VMs have `enable_console: false`) → `vm_panel` succeeds, only the requested VM's `enable_console` becomes `true` (verifies normal flow after restart)
- `vm_panel` called after `restart` if the `@game.reload.status` check is guarding the `update_column` (i.e. if `restart` set `game.status` to something other than `'active'` in the test) → `enable_console` remains `false` (verifies the mitigation guard)

If `restart` does not change `game.status` (the plan doesn't specify it does), document that the race mitigation relies on the **policy** guard (`vm_panel?` checks `record.status == 'active'`), and add a test that confirms a post-restart request is denied by policy.

### TEST-v4-9: `vm_status` Still Does Not Test Deleted-VmSet Response Shape (TEST-v3-3 Partially Unresolved)

**Severity:** Medium
**Phase:** 4.6

**Finding:**
TEST-v3-3 recommended splitting the `vm_status` test cases to cover the deleted-VmSet scenario:

> "`vm_status` (VmSet deleted post-creation): same shape as VM-free — confirm `vm_activation_mode` key is present"

The Revision 5 additions to Phase 4.6 include:

> "VM-free mission (no VmSet) → `activated: false`, `vms: []`"

This is the nil-`vm_set_id` case. The deleted-VmSet case — where `game.vm_set_id` is non-nil but `VmSet.find_by` returns nil — is still absent from the test list.

These are two distinct code paths. The controller's `vm_status` action (implicitly) does:

```ruby
vm_set = VmSet.find_by(id: @game.vm_set_id)
if vm_set
  render json: { ..., vm_activation_mode: ..., vms: [...] }
else
  render json: { activated: false, vms: [] }
end
```

The `else` branch is taken in both the nil-`vm_set_id` case and the deleted-VmSet case, but in the deleted-VmSet case `@game.vm_activation_mode` is still readable from the game's associated mission. A regression that accidentally omits `vm_activation_mode` from the `else` branch would break the HUD's ability to distinguish "VM-free mission" from "lazy mission with deleted VmSet" — the exact distinction highlighted in TEST-v3-3.

**Recommendation:**
Add explicitly to Phase 4.6:

- `vm_status` (game has `vm_set_id` set, but the VmSet record has been deleted): response contains `activated: false`, `vms: []`, AND `vm_activation_mode: 'lazy'` (or whatever the mission's mode is). This is distinct from the VM-free case where `vm_activation_mode` may be `'eager'`.

### TEST-v4-10: `finish` Has No Test for `game.update!` Exception — Unhandled 500 Is Undocumented

**Severity:** Medium
**Phase:** 4.6

**Finding:**
TEST-v3-8 identified that the `finish` action has no rescue block around its transaction. Revision 5 adds the `finish` scoring callback assertion (Phase 7.4) but does not add any test for the failure case. The plan still has no test that covers:

- `game.update!` raises `ActiveRecord::RecordInvalid` → transaction rolls back, VmSet is NOT relinquished, `RelinquishVmSetJob` is NOT enqueued, response is 500 (or, if a rescue block is added, a documented error JSON).

The `assert_enqueued_with` assertion added for the happy path (Phase 7.4: "`RelinquishVmSetJob` enqueued OUTSIDE transaction") implicitly verifies the job is enqueued, but does not test the absence of the job on failure.

More specifically, the plan states "`RelinquishVmSetJob` enqueued OUTSIDE transaction" — this phrasing implies the job is enqueued in an `after_commit` callback or after the transaction block closes. If `game.update!` fails inside the transaction, the transaction rolls back, `after_commit` does not fire, and the `perform_later` call after the block should also not be reached. **Is there a test that asserts `RelinquishVmSetJob` is NOT enqueued when `game.update!` fails?** The plan's test list does not include one. `assert_no_enqueued_jobs` in the failure scenario would pin this behaviour.

**Recommendation:**
Add to Phase 4.6:

- `finish` (game has a stubbed validation that causes `update!` to raise `ActiveRecord::RecordInvalid`): response is `500` (or a documented error JSON); `RelinquishVmSetJob` is NOT enqueued (assert via `assert_no_enqueued_jobs`); `GameCompletionScoringJob` is also NOT enqueued (the `after_commit` does not fire on rollback).

Alternatively, if the plan adds a `rescue` block to `finish` (recommended to prevent a 500 surfacing to the Phaser client), specify the expected error response shape and add a test that asserts it.

### TEST-v4-11: Concurrent `extend_vms` (Rapid Double-POST) Has No Server-Side Test

**Severity:** Medium
**Phase:** 4.4.2 / 4.6

**Finding:**
The plan adds a client-side `vmsActivating` flag (Phase 4.4.2) to prevent double-POSTs to `extend_vms`. The Revision 5 addition also covers the sequential idempotency case ("Second call (already activated): extends `activated_until`; does NOT redispatch activate"). However, the client-side flag is a UI guard, not a server-side guarantee. Two concurrent requests can bypass it (e.g. a browser reload, a network retry, or a client that does not implement the flag correctly).

The server-side handling of two simultaneous `extend_vms` requests is not tested. The concern is not sequential idempotency (already addressed in Revision 5) but simultaneous execution:

- Request A reads `vm_set.activated == false` → proceeds to dispatch activate to all VMs.
- Request B, processed concurrently, also reads `vm_set.activated == false` → also dispatches activate to all VMs.
- Both requests succeed; each VM receives two `activate` commands.

The VM infrastructure may handle duplicate activations gracefully, but this is not guaranteed and is untested. `DispatchVmCtrlService.ctrl_vm_async` dispatching twice to the same VM in the same second is a real-world possibility given Puma's multi-threaded request handling.

**Recommendation:**
Add to Phase 4.6:

- `extend_vms` (concurrent calls, optimistic lock or database guard): if the plan does not add a `WITH (EXCLUSIVE)` lock or an `update_columns` optimistic check, explicitly state in the test comment that concurrent activation is deferred and that the client-side guard is the only protection. At minimum, add a test that simulates two sequential `extend_vms` calls within the same test and asserts that `DispatchVmCtrlService.ctrl_vm_async` was called at most `vm_set.vms.count` times (not `2 × vm_set.vms.count`).

If `vm_set.update_columns(activated: true, ...)` is only called once (the first call) and the second call sees `vm_set.activated == true` and enters the "already activated" branch (which only extends `activated_until`), the existing idempotency test implicitly covers the benign concurrent case. The plan should state this explicitly: "The `activated` flag is the deduplication sentinel; two concurrent requests where the first has committed `activated: true` before the second reads it are handled by the idempotency branch."

### TEST-v4-13: `vm_set_panel` Test Cases Are Less Detailed Than `vm_panel` — Asymmetric Coverage

**Severity:** Low
**Phase:** 1.7

**Finding:**
Phase 1.7 specifies the following `vm_set_panel` functional tests:

- Valid request → redirects to Hacktivity vm_set URL with `embedded=1`
- Game has no `vm_set_id` → 404
- `vm_set.sec_gen_batch` is nil → 404
- `hacktivity_mode? = false` → 404
- Redirect URL is scoped to the correct event/batch/vm_set

This is a reasonable list, but it is missing several cases that `vm_panel` covers:

1. **Unauthenticated request → 401/redirect**: Present for `vm_panel`, absent for `vm_set_panel`. This is a generic authentication test and should apply identically to both actions.
2. **Admin access**: Specified for `vm_panel`, absent for `vm_set_panel`.
3. **Non-owner access (another player's game id)**: Specified for `vm_panel`, absent for `vm_set_panel`.
4. **`sec_gen_batch.event` being nil**: The `sec_gen_batch` nil guard is tested, but `sec_gen_batch.event` nil is not (ARCH-v4-1 identified this as a NoMethodError path). Present by implication in `vm_panel` via the redirect URL test, absent for `vm_set_panel`.

This asymmetry is unlikely to reflect intentional design — the two actions share essentially the same early-return logic and authorisation model. The coverage gap leaves `vm_set_panel` with weaker protection in tests than `vm_panel`.

**Recommendation:**
Mirror the `vm_panel` authentication and authorisation test cases onto `vm_set_panel` in Phase 1.7. Specifically add:

- Unauthenticated → 401/redirect (not 404)
- Non-owner → Pundit denied (403/redirect)
- Admin → allowed
- `vm_set.sec_gen_batch.event` is nil → 404, no redirect
